# 投資支援プラットフォーム — CLAUDE.md

## プロジェクト概要

投資判断の支援システムを構築中。テクニカル分析・ファンダメンタルズ分析・仮説検証・心理規律の4軸で、投資判断の精度を上げていくためのWebアプリケーション。

## 設計判断の経緯と理由（なぜこの設計になったか）

### 出発点
- ユーザーは投資・プログラミングともに知識ゼロの状態からスタート
- 最初は「テクニカル分析ダッシュボード」として開始
- 早い段階で「ダッシュボードと言っても、買い・売りがわかればよい。今は最低限」という明確な要求が出た
- これにより、情報を網羅的に並べるダッシュボード型から、**判定結果を最優先表示する「シグナル型」**に設計方針を転換

### Signal Engine の設計判断
- **多数決の原理を採用した理由**: 単一指標（RSIだけ等）は「ダマシ」が多い。7つの独立した指標が同じ方向を向いたときに確度が上がるという考え方が根底。複合スコアリングの根拠
- **タイムフレーム分割の理由**: デイトレードと中長期では見るべき指標が根本的に異なる。デイトレではVWAPやモメンタムが重要だが長期では無意味。長期ではSMA200やドローダウンが重要だがデイでは遅すぎる。同じエンジンで全てをカバーすると判定精度が落ちるため3モードに分離した
- **レジーム検出を入れた理由**: ADXでトレンド/レンジを自動判定し、レンジ相場ではトレンド系指標のスコアを0.7倍に減衰させる。これは資料の中で「ADXでトレンド/レンジを判別し戦略を動的にスイッチする」と記述されている考え方を実装したもの
- **SMAトレンドとMACDの重みが高い理由**: トレンドフォロー（順張り）が短期取引で最も再現性が高い戦略とされているため。cis氏の哲学「上がるものは上がる」との整合

### Research Lab の設計判断
- **きっかけ**: ユーザーが「AI投資術」という記事を見つけ、ニュース→セクター→銘柄の段階的分析アプローチに興味を持った
- **記事への評価（厳しくフラットに評価した結果）**:
  - 合理的な部分: トップダウン分析は機関投資家が使う王道。中長期との相性は良い
  - 問題点: デイトレードには向かない（分析中に変化は織り込み済み）、AIの株価予測は信用できない（もっともらしい文章を生成しているだけ）、生存者バイアスの罠
  - 結論: 「予測ツール」ではなく「リサーチ補助ツール」として割り切れば、中長期の銘柄スクリーニング補助として実用性がある
- **予測機能を排除した理由**: AIが「この株は上がる」と言っても予測精度に根拠がない。法的リスクもある。AIの役割は「情報整理と調査の出発点」に徹し、最終判断は必ずユーザー自身が行う設計

### 仮説ジャーナルの設計判断
- **ユーザーからの提案**: 「都度都度おれに（AIが）仮説を立てさせ、その検証をさせることで精度は高められるのでは？」
- **重要な修正**: AIが精度を上げるのではなく、**ユーザー自身の判断力を磨くため**のが本質。AIは情報整理と検証の補助をするだけ。仮説の蓄積と振り返りを通じて「どんなパターンの分析が当たりやすいか」を学ぶのはユーザー自身
- **的中率トラッキングの意味**: 単に当たった/外れたを記録するだけでなく、「なぜ当たったか」「何を見落としたか」を言語化して蓄積する。これが再現性のあるパターン認識につながる
- **セクター別集計の意味**: ユーザーが得意/不得意なセクターを客観的に把握できるようにする

### 規律ツール群の設計判断
- **きっかけ**: プロジェクトファイルとして持ち込まれた8つの投資資料を分析した結果、全資料に共通する6つの核心原則を抽出した。「読む知識」ではなく「使えるツール」として実装する判断
- **リフレーミング回診**: 全資料で繰り返し登場する「もし今持っていなかったら、今の価格で買うか？」を自動化。アンカリング（買値への固執）を仕組みで打破する
- **バイアス診断**: 確証バイアス・自信過剰・損失回避等の診断項目は、資料から具体的な症状記述を抽出して構成。月1回の定期実施を推奨設計
- **ケリー基準計算機**: 「一度に賭ける額を意見ではなく数式で決める」という原則の実装。ハーフケリー推奨は、推定誤差による実運用リスクへの対処
- **IPS**: 「冷静な今の自分が、パニック時の自分に残す遺言」という概念。全資料が損切りルールの事前設定を最重要視している

### UI/UXの決定経緯
- 初期バージョンは文字サイズ9-11pxで構築 → ユーザーから「文字が小さくて見にくい」のフィードバック → ベース14px、Noto Sans JPに変更
- Signal EngineとResearch Labが別ファイルだった → ユーザーから「行き来できるインターフェースがほしい」 → 1アプリ4タブに統合
- 情報量を詰め込むダッシュボード型 → 「買い・売りが一目でわかること」が最優先 → 判定結果を画面中央に大きく表示、詳細は展開式

### 精度向上の戦略
- **現在の限界**: シミュレーションデータで動いているため、実際の精度は未検証
- **次にやるべきこと**: 実データAPI接続 → バックテスト（過去データでスコアリング戦略の収益を計算）→ パラメータ最適化（どの指標の重み配分が最も収益が高いか）
- **中期目標**: マルチタイムフレーム確認（日足+週足が同方向なら信頼度UP）、ファンダメンタルズ統合（PER/PBR/決算データ）、センチメント分析
- **仮説ジャーナルの蓄積が鍵**: 50件、100件と仮説が溜まれば「自分はテック株の分析が得意で、金融セクターは苦手」等のパターンが見えてくる。これがユーザー固有のエッジになる

### Claude Code 移行の判断理由
- 1つのJSXファイルに約740行が集中し、機能追加・修正のたびに全体書き直しのリスク
- バックテストにはローカルでNode.jsスクリプトを実行する必要がある
- 実データAPI接続にはAPIキー管理やバックエンドが必要
- ファイル分割により、指標の追加や修正が局所的に完結する
- ただし、単に構造だけ整えても使えなければ意味がない。そのためプロトタイプで機能を確定してから移行する判断とした

## ユーザーについて

- 投資知識は初学者レベル。専門用語には平易な日本語の解説を必ず付ける
- プログラミング知識もゼロからスタート。環境構築等のサポートが必要
- 厳しく・フラットなフィードバックを好み、粉飾せず可能性として検討する姿勢
- UIの好み: 文字は大きく（ベース14px以上）、ダークテーマ、最小限の情報で「買い/売り」が一目でわかること

## 4つの機能モジュール

### 1. Signal Engine（テクニカル分析）
シミュレーションデータ（将来は実データ）に対し、複数のテクニカル指標で複合スコアリングを行い、買い/売り判定を出す。

#### 3つのモード
- **デイトレード**: 5分足ベース、6指標（VWAP乖離, 短期モメンタム, RSI(7), ストキャス(5,3), 出来高スパイク, 足型パターン）
- **スイングトレード**: 日足ベース、7指標（SMAトレンド, MACD, RSI(14), ボリンジャーバンド, ADX, 出来高, ストキャスティクス）
- **中長期投資**: 日足長期、7指標（SMA200, トレンド傾き, MACD長期(26/52/18), RSI(21), ADX, ドローダウン, 出来高トレンド）

#### スコアリング仕様
- 各指標: -100〜+100のスコアを算出
- 各指標にweight（重み%）を設定、加重平均で総合スコア算出
- 判定閾値: +45以上=強い買い / +20~44=買い / +5~19=やや買い / -4~4=様子見 / -5~-19=やや売り / -20~-44=売り / -45以下=強い売り
- 信頼度: 7指標中、同方向(|score|>10)が70%以上=高、40%以上=中、それ以下=低

#### マーケットレジーム検出
- ADX>30: 強いトレンド（順張り指標の信頼性高い）
- ADX 20-30: 中程度のトレンド
- ADX<20: レンジ相場 → 全スコアに0.7倍の減衰をかける（トレンド系指標の信頼性低下）

#### 各指標に付随する情報
- 「📖 この指標とは？」: 初心者向けの平易な解説
- 「⚡ 判定根拠」: なぜそのスコアになったかの理由リスト
- 「📚 用語辞書」: その指標に関連する専門用語の解説

### 2. Research Lab（ファンダメンタルズ分析）
ニュースからセクター、銘柄候補まで段階的に絞り込む7ステップのリサーチワークフロー。

#### 7ステップ
1. **ニュース収集**: Claude API + web_searchで市場影響ニュース5つを取得。各ニュースに概要・影響度(高/中/低)・時間軸(短/中/長)
2. **深掘り**: ユーザーが選んだトピックの背景・経緯・3シナリオ(楽観/中立/悲観)・注目ポイント
3. **セクター分析**: 恩恵セクター3つ + 逆風セクター3つ。各セクター名・影響度・一行理由
4. **根拠検証**: 各セクターの因果関係・過去類似ケース・想定時間軸・リスク要因
5. **銘柄候補**: 指定セクターの企業5社。企業名/コード・事業概要・関連性・競争優位性・リスク（ダイレクトに銘柄含む）
6. **企業深掘り**: 1社の会社概要・業績トレンド・競合比較・事業環境要因・財務注意点
7. **仮説化**: 分析結果を仮説ジャーナルに記録（Step 5以降で表示）

#### Claude API仕様
- model: claude-sonnet-4-20250514
- max_tokens: 1000
- tools: web_search_20250305 を有効化
- system prompt: 「投資リサーチアシスタント。客観的事実のみ。投資推奨は絶対にしない」
- 各ステップは前ステップの結果をコンテキストとして渡す

#### UIフロー
- ステップは順番にアンロック（Step 1完了 → Step 2が表示）
- 各ステップにアクションボタンと結果表示エリア
- Step 2,5,6はユーザー入力フィールド付き
- ローディング中はボタン非活性 + アニメーション

### 3. 仮説ジャーナル
投資仮説を立て、時間経過後に検証し、的中率を追跡するシステム。蓄積した記録から「自分はどんな分析が当たりやすいか」を学ぶことが目的。

#### データ構造（1件あたり）
```
{
  id: number (Date.now()),
  thesis: string,          // 仮説の本文（例:「半導体セクターはAI需要拡大で中期的に上昇する」）
  sector: string,          // 対象セクター
  timeframe: string,       // 検証期間（"1ヶ月" | "3ヶ月" | "6ヶ月" | "1年" | "2年以上"）
  basis: string,           // 根拠（なぜそう考えたか）
  createdAt: ISO string,   // 作成日時
  status: string,          // "検証中" | "的中" | "一部的中" | "外れ"
  reviewedAt: ISO string,  // ステータス変更日時
  review: string,          // 振り返りメモ（何が想定と違ったか、次に活かすこと）
  lastVerification: string, // AI検証の結果テキスト
  lastVerifiedAt: ISO string // AI検証の実行日時
}
```

#### 保存仕様
- キー: "hypo-v2"
- 形式: JSON配列
- 永続化: localStorage（将来はSupabase等のDBに移行可能）
- 新しい仮説は配列の先頭に追加（逆順表示）

#### AI検証のプロンプト設計
「仮説の検証:「{thesis}」({sector}, {createdAt}設定, {timeframe})
根拠:{basis}
→最新状況、支持材料、否定材料、当初との乖離を分析」
- Claude APIのweb_searchツールを有効にして最新情報を取得
- システムプロンプト: 客観的事実のみ提供、投資推奨は絶対にしない

#### 的中率の計算ロジック
- 的中率 = (的中数 + 一部的中数 × 0.5) / (的中 + 一部的中 + 外れ) × 100
- 「検証中」は分母に含まない
- セクター別にも同じ計算を行い、得意/不得意セクターを可視化

#### UIフロー
1. 仮説一覧: ステータス別の色分け表示、根拠・検証結果の展開表示
2. 新規作成: thesis, sector, timeframe, basis の4フィールド入力
3. ステータス変更: 「的中」「一部的中」「外れ」ボタンでワンタップ切替
4. AI検証: ボタン押下でClaude APIに現在の状況を問い合わせ、結果を保存
5. 振り返りメモ: 各仮説にテキスト入力欄、リアルタイム保存
6. 成績タブ: 総数/検証中/的中/外れ/的中率のカード + セクター別集計

### 4. 規律ツール群
投資心理学の原則に基づき、感情による判断ミスを仕組みで防止する4つのツール。

#### リフレーミング回診
- 入力: 銘柄名、買値、現在値
- 自動計算: 含み損益（%と金額）
- 含み損時: プロスペクト理論の警告メッセージ表示
- 核心の問い: 「今持っていなかったら、今の価格で新たに買うか？」
- 回答: Yes（保有継続に合理性あり）/ No（売却検討を推奨）/ Maybe（追加リサーチ必要）
- 各回答に応じた解説・アドバイス表示
- 判断理由メモ欄
- 保存キー: "reframes-v1"、データ構造: {id, name, buyPrice, nowPrice, answer, reason, date}
- 過去の回診記録一覧表示（直近10件）

#### バイアス自己診断
- 14項目のチェックリスト（4カテゴリ）
  - 自信過剰バイアス: 4項目
  - 確証バイアス: 3項目
  - アンカリング・損失回避: 4項目
  - その他の危険信号: 3項目
- 各項目はチェックボックス、リアルタイム保存
- 診断結果: チェック数/14、バイアスリスクレベル（良好<30%/注意30-59%/危険60%+）
- 保存キー: "bias-checks-v1"、データ構造: {項目ID: boolean}のオブジェクト
- 月1回の定期実施を推奨

#### ケリー基準計算機
- 入力: 勝率(%)、平均利益率(%)、平均損失率(%)、総資産額(¥)
- 計算: K = (p × (B + 1) - 1) / B （B = 平均利益率 / 平均損失率）
- 出力: 期待値、フルケリー値、ハーフケリー値（推奨）、推奨投入額
- 期待値がマイナスの場合: 手法見直しの警告表示
- 公式の計算過程を数値入りで可視化
- ハーフケリーを推奨する理由の解説表示

#### IPS（投資方針書）
- Clause 1: 損切りライン（数値入力、推奨値: 短期5-8%/スイング8-15%/長期15-20%）
- Clause 2: 1銘柄ポジション上限（数値入力）
- Clause 3: リバランス頻度（選択式: 毎月/四半期/半年/年1回）
- Clause 4: エコノミックモート崩壊ルール（自由記述テキストエリア）
- Clause 5: 自分ルール（自由記述、ナンピン禁止・SNS遮断等）
- 全項目はリアルタイム自動保存
- 保存キー: "ips-v1"、データ構造: {stopLoss, maxPosition, rebalancePeriod, moatRule, customRules}
- 定次リフレーミング・リマインダーの案内表示

## 採用している投資の核心原則（docs/の8資料から抽出）

8つの投資資料は言い口は異なるが、繰り返し同じ6原則を説いている。これをシステムの設計根拠として採用。

1. **期待リターン10.4%**（日経75年平均、大数の法則）。2023年の28.2%等の短期高パフォーマンスは統計的異常値（アウトライヤー）に過ぎない。脳が小さなサンプルを全体像と勘違いする性質への対策。ドルコスト平均法で試行回数を稼ぎ大数の法則を味方にする
2. **プロスペクト理論**: 損失の痛みは利益の喜びの2倍以上。「含み益は早く確定したがる、含み損は認めたがらない（ディスポジション効果）」「買値に固執する（アンカリング）」が「利小損大」を招く。**意志力では克服不可能**→仕組み（逆指値、IPS、リフレーミング）で対処するしかない
3. **順張り哲学**（cis氏「上がるものは上がる、下がるものは下がる」）。押し目買いは「いつか戻るはず」という身勝手な仮説に基づく逆張りの一種として危険。cis氏自身がバリュー投資（逆張り）で2年で1000万円を溶かした後に順張りに転向した事実がある
4. **エコノミックモート**（バフェット流）。企業の堀（競争優位性＝コスト優位性、ブランド、流通網、エコシステム、キャッシュフロー）が重要。会計上の純利益は「役立つが、ひどい」（未実現利益や税制変更で歪む）。実質的なキャッシュ創出力を見る。「能力の輪」の内側にとどまり、理解できないビジネスには投資しない
5. **ケリー基準**: K=(p(B+1)-1)/B。期待値がプラスの手法を破滅せずに繰り返すための数学的裏付け。ハーフケリー推奨（推定誤差リスクの緩和）。2%ルール（1トレード最大損失を総資金の2%以内）と併用。50%損失からの回復には100%の利益が必要という「損失の非対称性」への対策
6. **ルールベース運用**: IPS投資方針書（パニック時の自分を縛る契約書）、逆指値（損切りの自動化で感情をバイパス）、ドルコスト平均法（タイミング判断を排除）、定期リバランス、リフレーミング（定次で「今買うか？」を問い直す）。感情に頼る投資から、システムによる投資への転換

## 絶対に守るべき制約

- **AIは予測しない**: 株価の方向性予測、購入推奨、売却推奨は一切行わない。法的リスクがある。AIの役割は情報整理・リサーチ補助に徹する
- **最終判断はユーザー**: AIの出力は「調査の出発点」。どれだけ高いスコアが出ても、買い/売りの最終決定はユーザー自身
- **免責事項の表示**: 全ページに「投資助言ではない」「シミュレーションデータ使用」「投資は自己責任」を明記
- **ナンピン（買い下がり）の警告**: 下がったから買い増す行為は「損失領域でのリスク追求」であり破滅リスク大。システム上でナンピンを推奨する機能は作らない

## 開発ルール

- 1ファイル300行以内を目安に分割する
- 新しい指標や機能を追加する際は、既存のスコアリングエンジンのインターフェースに合わせる
- UIの変更時はダークテーマ・14px以上の文字サイズを維持、フォントはNoto Sans JP + JetBrains Mono
- AIが投資推奨・株価予測をする機能は絶対に実装しない
- 免責事項を全ページに表示する
- 専門用語を使う場合は必ず平易な日本語の解説を付ける（ユーザーは初学者）
- 「買い/売り」の判定結果が最も目立つUI設計にする。詳細は展開式
- ナンピンを推奨する機能は作らない
- Research LabのClaude APIプロンプトには必ず「投資推奨はしない」を含める

## docs/フォルダの参考資料について

8つの投資資料がdocs/に格納されている。システムの設計根拠である6つの核心原則の出所。
機能追加や設計判断で迷った場合は、これらの資料を参照し、原則との整合性を確認すること。
内容は以下の通り:
- 期待リターンと大数の法則（28.2%の幻想 vs 10.4%の現実）
- プロスペクト理論と損切りの科学
- テクニカル分析完全ガイド（4つの取引スタイル、指標の組み合わせ戦略、マルチタイムフレーム分析）
- バフェット流の価値創出とエコノミックモート
- 心理的バイアス分析マニュアル（顧客タイプ別の罠と対策）
- 数理的資金管理（2%ルール、ポジションサイジング、ATRベースの損切り幅設定）
- 2026年市場転換期のロードマップ（日米金利差縮小、AI銘柄の収益化フェーズ、ROE/PER/PBR統合分析）
- 投資スタイル別選定ガイド（複利の力、能力の輪、20回のパンチカード）
